集群的节点出现了 NotReady 的情况，这里做一下记录

为什么会 Node NotReady?

一般节点出现 NotReady 有以下几种原因：

  • kubelet 进程停止
  • apiserver 进程停止
  • etcd 进程停止
  • kubernetes 管理网络 Down

如何处理 NotReady 的情况

  1. 查看集群区和 node 状态

    kubectl describe nodes

  2. SSH 对应的 node 节点，查看 kubelet 状态

  3. 查看日志 journalctl -u kubelet

疑问，为什么 Pod 没有重建？

最后通过重启 node 的 kubelet 服务，问题得到了解决，但是仍然有疑问，为什么 Pod 没有重新创建呢？

这是因为当节点发生 NotReady 的时候，不同的类型的资源，集群会有不同的处理方式：

  • Deployment: 会在其它节点创建新的实例，故障节点恢复后，kubelet 执行优雅删除，删除旧的 PodObj
  • DaemonSet: 会从 NodeLost 状态直接变成 Running 状态，不涉及重建
  • StatefulSet: statefulset 会从Unknown 状态变为 Terminating 状态，执行优雅删除，detach PV，然后执行重新调度与重建操作。
